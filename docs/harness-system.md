# Harness 文件功能说明

ego 项目实施**分层脚手架（Hierarchical Harness）**规范。为了防止上下文污染并控制 AI Agent 的执行边界，脚手架按作用域被严格划分为“全局集成层”与“局部沙盒层”。

## 1. 全局集成层 (项目根目录 `.harness/`)

根目录下的脚手架专门用于管理**跨端（前端+后端）**或**跨模块**的联调与集成。严禁在此处记录任何单一模块内部的开发细节。

- **`integration_feature_list.json`**：**全局集成任务池**。以强类型 JSON 格式记录需要多端对齐的全局 Feature 清单及验收标准，是统筹跨端联调的唯一指令源。
- **`integration-progress.md`**：**全局状态与记忆交接**。记录仓库维度的联调进度、API 契约 (`proto`) 对齐状态及全局级的系统卡点（Blockers）。
- **`clean-state-checklist.md`**：**全局准出校验**。定义 Agent 在宣布一个全局 Feature 完成前，必须在项目根目录执行的强制检查流（例如：`make proto` 是否已重新编译、`make check` 全局扫描是否通过、前后端集成测试是否全绿）。

---

## 2. 局部沙盒层 (各业务子模块，以 `internal/writing/` 为例)

在后端 DDD 架构或独立的前端 (`client/`) 目录中，每个业务模块均作为一个自治的开发沙盒。其专属的脚手架被物理划分为“架构契约层”与“执行记忆层”：

### 2.1 模块根目录 (架构与契约层)
此类文件为低频修改项，主要向 Agent 灌输模块的设计理念与边界规范。

- **`AGENTS.md`：规则与路由**。定义 Agent 在当前沙盒内的工作流与硬约束，明确模块级职责及专属验证命令。
- **`ARCHITECTURE.md`：内部架构设计**。固化本模块的 DDD 设计，明确定义聚合根、实体、值对象与应用层编排，防止大模型退化为传统 CRUD 思维。
- **`CONTRACT.md`：对外服务契约**。声明本模块向其他兄弟模块开放的公共接口 (Port/Interface)、只读视图或领域事件。跨模块交互时仅限阅读此文件，严禁检索底层实现代码。

### 2.2 模块内的 `.harness/` 目录 (执行与记忆层)
此类文件为高频读写项，是 Agent 在本模块内执行具体开发任务的动态工作台。

- **`feature_list.json`**：**局部任务调度池**。记录纯属本模块内部的开发节点清单。Agent 必须依据此文件改变开发行为，严禁在此混入跨模块联调任务。
- **`progress.md`**：**局部记忆与状态交接**。记录当前模块的最新代码修改进度、底层逻辑卡点与下一步计划，确保单一模块内跨会话上下文的绝对连续，防止 AI 失忆。
- **`clean-state-checklist.md`**：**局部准出校验**。定义 Agent 终止当前模块开发会话前必须完成的强制自查（如局部 `go test ./...` 运行通过、调试打印代码清理），确保该沙盒提交的代码始终处于健康状态。
